## 1. 产品概述

### 1.1 产品愿景

打造一个完全由个人掌控的、能够深度理解用户习惯与需求的智能助手平台。它不仅是文件管理、歌单、待办等工具，更是一个主动感知、自主决策、持续进化的数字伙伴，通过捕获用户日常行为，在恰当的时机提供恰好的智能服务。

### 1.2 产品定位

一个私有化部署的个人 AI 操作系统：整合文件、媒体、事务、知识管理，所有数据自主掌控。

两层智能架构：

薄工具层：提供传统功能（文件管理、音乐播放、待办清单、文档阅读等）。

厚 AI 层：基于事件流、记忆网络、意图识别和自动化引擎，实现主动服务。

### 1.3 目标用户

对数据隐私有高要求的极客、知识工作者、创作者。

希望用 AI 提升日常数字生活效率的个人。

初期为单一用户（作者本人），但架构上预留多用户扩展能力。

3. 用户故事与使用流程
   3.1 典型的一天
   早上 8:30：AI 基于过往开机时间，在你靠近电脑前已将今日待办和推荐的晨间音乐准备好。

上午 10:15：在专注工作 45 分钟后，AI 检测到你开始频繁切换窗口，弹出一条温柔提醒：“你已连续专注工作 45 分钟，要休息 5 分钟并来点轻松音乐吗？”

下午 14:00：正在准备季度汇报 PPT，AI 感知到你频繁搜索“Q3 数据”，主动从文件库中调出相关 Excel 和图表，并提供“一键插入 PPT”的建议。

晚上 22:30：AI 总结今天完成的事项，并提醒明天是家人好友的生日（来自聊天记录/日历抽取）。

3.2 核心用户故事
作为用户，我可以通过一个统一的界面管理所有文件（文档、音乐、视频），无需关心底层存储。

作为用户，我希望通过自然语言告诉 AI：“帮我创建一个适合雨天听的歌单”，AI 能自动从我的音乐库里挑选并生成。

作为用户，当我浏览文档时，只需说一句“下周前完成这个评审并关联此文件”，系统就能自动创建待办并建立关联。

作为用户，我安装桌面客户端后，AI 能逐渐了解我的工作习惯，并在我需要时主动提供帮助，而不是被动等待指令。

作为用户，我能够完全控制自己的数据，随时查看被采集的事件、删除记忆，并确保敏感信息不上传。

4. 详细功能需求
   4.1 前端界面
   4.1.1 整体布局
   左侧导航栏：文件、音乐、文档、视频、待办事项、智能推荐、AI 记忆地图。

顶部全局搜索：支持关键词搜索和语义搜索，可限定类型。

全局悬浮 AI 助手按钮：点击弹出对话面板，支持流式回复、上下文感知。

通知中心：集中展示主动建议、任务完成、系统提醒等卡片。

4.1.2 文件管理页
网格/列表视图切换，支持文件夹树状导航。

拖拽上传（预签名 URL 直传 MinIO），上传进度显示。

右键菜单：移动到歌单、设为待办附件、AI 摘要、分享链接等。

标签管理：添加、编辑、按标签过滤。

文件预览：图片、音频、视频内嵌播放；文档支持在线预览（通过 onlyoffice 或浏览器内置）。

回收站功能，支持还原或彻底删除。

4.1.3 歌单页
创建、编辑、删除播放列表。

从文件库拖拽添加歌曲，支持排序。

播放器：播放/暂停、上一首/下一首、进度条、音量控制、播放模式。

专辑封面显示，歌词展示（如果文件包含）。

AI 推荐模块：“此刻推荐”的动态播放列表。

4.1.4 文档中心
已上传文档列表，支持按类型、标签筛选。

点击文档进入预览（Markdown 直接渲染，PDF/Word 转 HTML 预览或集成 onlyoffice）。

内嵌 AI 问答：可针对当前文档提问、生成摘要、翻译内容。

版本历史查看（若开启版本管理）。

4.1.5 视频库
视频卡片式画廊，自动生成缩略图。

点击播放，流式传输（HLS/DASH 或直接 Range 请求）。

标签、评分，AI 自动识别内容并打标。

4.1.6 待办事项页
看板视图（未开始/进行中/已完成）和列表视图。

支持拖拽改变状态、排序。

关联文件：显示附件预览。

自然语言快速创建：“明天下午 3 点开会 @urgent”。

智能提醒：到期通知、重复任务、AI 根据上下文建议的待办。

4.1.7 AI 对话面板
支持多轮对话，显示历史记录。

根据当前页面注入上下文（如在文件页问“这个文件夹里哪些是上周修改的”）。

输出可交互卡片（歌单、待办、文件列表等）。

支持语音输入（P2）。

4.1.8 智能推荐与发现
“今日推荐”板块：歌单、文档、视频，基于习惯和当前情境。

推荐理由简要说明（如“根据你最近常听的 Jazz 风格生成”）。

4.1.9 记忆与画像管理
记忆地图可视化：以图谱形式展示 AI 对你的认知（兴趣、常用工具、偏好等）。

可手动修正或删除记忆节点。

遗忘功能：按时间范围或主题擦除相关数据。

4.2 后端服务
4.2.1 对象存储 (MinIO)
Bucket 规划：files（通用文件）、music、videos、documents、thumbnails、archives。

文件上传采用预签名 URL，前端直传，减轻后端压力。

文件元数据（原始名、大小、MIME、标签、收藏）存 MySQL。

生命周期策略：缩略图临时桶自动过期，原文件长期保留。

4.2.2 关系数据库 (MySQL)
核心表：

users：用户信息。

files：统一文件元数据表，通过 bucket 字段区分类型，tags 用 JSON。

playlists / playlist_items：歌单及与文件的关联。

todos：待办事项，关联用户、状态、优先级、截止时间、附件。

user_events：记录聚合后的用户行为事件（应用使用、时长、会话等）。

user_profile：动态更新的用户画像（兴趣向量、工作模式、响应风格偏好）。

automation_rules：用户自定义的自动化规则。

索引优化：针对常用查询添加复合索引，对 tags JSON 字段使用虚拟列索引。

4.2.3 向量数据库 (Qdrant / Milvus)
集合设计：

file_memory：文件内容的语义向量（用于语义搜索）。

conversation_memory：对话片段向量，保留近期历史。

context_snapshots：用户情境快照向量（当前活动、时间、情绪等）。

user_interest：兴趣向量（定期更新）。

相似性搜索：结合过滤器（文件类型、时间范围）进行语义检索。

嵌入模型：本地运行通用嵌入模型（如 BGE-M3），保证隐私。

4.2.4 事件接收与流处理
事件采集 API：接收客户端加密上报的事件，解密后写入 Redis Streams。

聚合服务：消费流数据，按会话切分，提取特征（专注度、情绪倾向、活跃应用）。

情境快照生成器：定时（5-10 分钟）生成当前状态的自然语言描述，向量化存入 Qdrant。

模式挖掘引擎：每日离线分析 MySQL 中的事件数据，挖掘高频工作流、周期习惯、异常行为，存入知识图谱或规则表。

4.2.5 AI Agent 引擎 (LangChain / 自研)
意图识别与工具调用：

定义工具集：search_files、create_playlist、add_todo、recommend_content、summarize_file、exec_automation 等。

LLM 负责解析用户输入为工具调用链，支持多步规划。

记忆管理：

短期记忆：最近 N 轮对话存向量库，按时间衰减。

长期记忆：定期从情境和对话中抽取事实，存入知识图谱（Neo4j 或图结构表）。

情境感知：

在 Agent 处理任务时，自动注入 current_context（来自事件流的最新冠注）。

主动服务触发器：

规则引擎 + 预测模型：当情境满足条件时，生成通知或建议卡片。

如：IF 专注时间 > 45分钟 THEN 建议休息并播放放松音乐。

多步任务执行与反思：

复杂任务规划（如整理项目文件）拆解为子任务，逐步执行，允许用户确认高风险动作。

自我反思：执行后评估结果，优化下次策略。

4.3 用户事件采集 App
4.3.1 功能概述
一款跨平台（优先 Windows/macOS）桌面客户端 + 浏览器扩展，静默收集用户行为元数据。

所有数据本地缓冲、加密后上传到私有后端，确保隐私。

4.3.2 采集的事件类型
桌面应用事件：窗口切换（应用名、窗口标题）、使用时长。

输入特征：打字速度、删除率、鼠标移动热区（不记录具体按键内容）。

文件活动：文件打开/编辑/保存操作（路径脱敏或本地映射为标签）。

剪贴板监控：文本长度、类型（URL/文本/图片），不上传具体内容。

浏览器活动：当前页面域名、标题、停留时间，搜索关键词（脱敏）。

系统状态：开机/关机、锁屏、电量、网络变更。

媒体播放事件：播放/暂停、歌曲名（若使用系统播放器）。

4.3.3 隐私与安全
客户端本地 SQLite 缓冲，数据加密存储。

上传前在本地进行脱敏处理：替换具体文件名、路径为抽象标签，剪贴板内容不上传。

端到端加密传输。

用户可随时暂停采集、查看已采集数据、删除单条或批量清除。

绝不采集：密码、聊天内容、银行卡信息、摄像头画面。

4.3.4 设置面板（客户端内）
开关各采集项（如“记录应用使用，但不记录浏览器 URL”）。

数据保留天数（本地/云端）。

查看最近 24 小时采集摘要。

4.4 自动化与任务引擎
规则创建界面（P2）：If-This-Then-That 可视化编辑。

预设规则模板：如“新建文档时自动生成摘要”、“完成待办后自动归档关联文件”。

AI 建议规则：根据你的习惯推荐可自动化的工作流。

执行日志：所有自动化运行记录，可回溯。

5. 非功能性需求
   5.1 性能
   文件预览（图片/音频/视频）加载延迟 < 1 秒。

AI 对话回复首字流式输出时间 < 2 秒（本地模型）。

事件采集 CPU 占用 < 2%，内存 < 150MB。

支持 10 万文件量级下全文搜索毫秒级响应。

5.2 安全
用户身份认证 JWT，所有 API 需鉴权。

MinIO 存储桶私有化，通过后端生成临时访问 URL。

所有敏感配置通过环境变量注入，不硬编码。

前端与后端、客户端与后端通信均使用 HTTPS/WSS。

定期漏洞扫描和安全更新。

5.3 可用性与可维护性
提供 Docker Compose 一键部署脚本。

详细的部署文档和配置说明。

后端日志分级，可快速定位问题。

数据库定期自动备份（MinIO 中保留备份文件）。

5.4 可扩展性
微服务架构趋势：早期可整合在单一后端，但按模块划分清晰的服务包，未来可拆分。

支持多模型切换（OpenAI / 本地 Llama / Claude），通过配置更换 LLM 提供商。

插件化工具集：新增功能只需注册 Tool 并实现接口。

7. 数据隐私与伦理条款
   数据最小化原则：仅采集实现功能所必需的最少数据。

本地化处理：尽可能在客户端完成脱敏、特征提取，云端仅存储不可逆的元数据。

透明与控制：前端提供完整的数据日志查看界面，支持数据导出和删除。

无第三方传输：所有数据只存储于用户个人控制的服务器，绝不出售或用于广告。

遗忘机制：任何记忆数据都可通过界面或 API 永久删除，不留备份。

9. 风险与依赖
   隐私合规风险：确保事件采集符合用户心理预期，通过详细的隐私说明和可控开关降低担忧。

本地模型性能：若使用本地 LLM 推理，硬件要求较高，需评估个人服务器配置。初期可部分使用云端 API，后续迁移本地。

数据量增长：事件采集会产生海量数据，需设计合理的聚合与归档策略，防止存储爆炸。

跨平台兼容性：桌面客户端需处理 Windows/macOS/Linux 的系统 API 差异。

10.3 用户事件表 (user_events)

```sql
CREATE TABLE user_events (
id INT AUTO_INCREMENT PRIMARY KEY,
user_id INT NOT NULL,
event_type VARCHAR(50) NOT NULL,  -- 'app_switch','input_stats','file_action','web_visit'等
session_id VARCHAR(64),
payload JSON,                     -- 存放经过聚合脱敏的事件详细数据
recorded_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
FOREIGN KEY (user_id) REFERENCES users(id),
INDEX idx_user_time (user_id, recorded_at)
);
```
